# GCM MCP Server — End-to-End Setup Guide for IBM Bob

This guide takes you from zero to a working IBM Bob + GCM MCP integration.
Two roles are involved — an **Admin** who sets up the server once, and **Users** who configure their Bob client.

---

## Prerequisites

### Admin needs

| Requirement | Details |
|-------------|---------|
| A Linux server with Docker | Any machine reachable from the user's network. Does **not** need to be the GCM server. |
| SSH access to the GCM server | To extract the OIDC client secret from Kubernetes |
| GCM admin credentials | `gcmadmin@gcm.local` account (or another admin account) |
| Port 8002 open | In the Docker host's firewall / security group |

### User needs

| Requirement | Details |
|-------------|---------|
| IBM Bob installed | Cloud desktop or desktop app |
| Server URL | Provided by the admin after setup (e.g., `http://10.0.0.5:8002/sse`) |
| API key | Generated by the admin specifically for each user |

---

## Part A: Server Setup (Admin — one time)

### A1. Extract OIDC Credentials from the GCM Kubernetes Cluster

The MCP server authenticates to GCM via Keycloak (GCM's built-in identity provider). The credentials are stored as a Kubernetes secret on the GCM server.

```bash
ssh root@<gcm-server-ip>
export KUBECONFIG=/etc/rancher/k3s/k3s.yaml

# Get Client ID (usually "gcmclient")
kubectl get secret oidc-client-secret -n gcmapp \
  -o jsonpath='{.data.CLIENT_ID}' | base64 -d && echo

# Get Client Secret
kubectl get secret oidc-client-secret -n gcmapp \
  -o jsonpath='{.data.CLIENT_SECRET}' | base64 -d && echo
```

Save both values — you need them in step A3.

---

### A2. Pull the Container Image

SSH into the machine that will host the MCP server:

```bash
ssh root@<mcp-server-host>
docker pull ghcr.io/ibm/gcm-mcp-server:latest
```

The image supports both `linux/amd64` and `linux/arm64`.

---

### A3. Run the Container

```bash
docker run -d \
  --name gcm-mcp-server \
  --restart unless-stopped \
  -p 8002:8002 \
  -v gcm-mcp-data:/data \
  -e GCM_HOST="<gcm-server-ip>" \
  -e GCM_USERNAME="gcmadmin@gcm.local" \
  -e GCM_PASSWORD="<gcm-admin-password>" \
  -e GCM_CLIENT_ID="gcmclient" \
  -e GCM_CLIENT_SECRET="<from-step-A1>" \
  ghcr.io/ibm/gcm-mcp-server:latest
```

**What each parameter does:**

| Parameter | Purpose |
|-----------|---------|
| `-p 8002:8002` | Exposes the MCP SSE transport on port 8002 |
| `-v gcm-mcp-data:/data` | Persistent volume — API keys survive container restarts and upgrades |
| `GCM_HOST` | IP or hostname of the GCM appliance (e.g., `9.30.252.230`) |
| `GCM_USERNAME` | GCM admin account used by the server to call GCM APIs |
| `GCM_PASSWORD` | Password for that account |
| `GCM_CLIENT_ID` | Keycloak OIDC client ID (from step A1, usually `gcmclient`) |
| `GCM_CLIENT_SECRET` | Keycloak OIDC client secret (from step A1) |

> **Security note:** Users never see these credentials. The MCP server holds them server-side and uses them transparently. Users only authenticate to the MCP server itself via API keys (step A5).

---

### A4. Verify the Server is Running

```bash
curl http://localhost:8002/health
```

Expected response:

```json
{
  "status": "ok",
  "server": "GCM MCP Server",
  "version": "1.0.0",
  "transport": "sse",
  "auth_required": true,
  "active_keys": 0,
  "services": [
    "usermanagement", "tde", "assetinventory", "discovery",
    "policy", "policyrisk", "audit", "integration",
    "notifications", "clm", "config"
  ]
}
```

- `auth_required: true` — API key enforcement is active
- `active_keys: 0` — no user keys yet (you create them in the next step)
- `services` — all 11 GCM services are discovered and ready

---

### A5. Generate an API Key for Each User

Run this from the MCP server host (admin endpoint is localhost-only):

```bash
curl -s -X POST http://localhost:8002/admin/keys \
  -H "Content-Type: application/json" \
  -d '{"user": "alice@ibm.com"}'
```

Response:

```json
{
  "key": "a3f8e9b1c4d7e6f2...full-64-character-hex-string...",
  "user": "alice@ibm.com",
  "created": "2026-02-21T10:00:00Z",
  "key_prefix": "a3f8e9b1"
}
```

> **Important:** Copy the `key` value immediately and send it to the user securely (encrypted email, Slack DM, in person). The server only stores a SHA-256 hash — the raw key is shown **exactly once** and can never be retrieved again.

Send each user:
1. The **server URL**: `http://<mcp-server-host>:8002/sse`
2. Their **API key** (the 64-character hex string above)

## Part B: Client Setup — IBM Bob (Each User)

### What you need from the admin

1. **Server URL** — e.g., `http://9.30.147.112:8002/sse`
2. **API key** — the 64-character hex string the admin generated for you

---

### B1. Bob's MCP Settings File Location

Bob stores all MCP server configurations in a single JSON file:

| Operating System | File Path |
|-----------------|-----------|
| **macOS** | `~/Library/Application Support/IBM Bob/User/globalStorage/ibm.bob-code/settings/mcp_settings.json` |
| **Linux** | `~/.config/IBM Bob/User/globalStorage/ibm.bob-code/settings/mcp_settings.json` |
| **Windows** | `%APPDATA%\IBM Bob\User\globalStorage\ibm.bob-code\settings\mcp_settings.json` |

---

### B2. Option A — Ask Bob to Configure Itself (Recommended)

You can paste this prompt directly into Bob and it will write the settings file for you:

```
I want to configure the GCM MCP server in my IBM Bob settings.

Please update my MCP settings file at:
~/Library/Application Support/IBM Bob/User/globalStorage/ibm.bob-code/settings/mcp_settings.json

Add the following MCP server entry (do not remove any existing entries):

Server name: gcm-mcp-server
Type: sse
URL: http://<mcp-server-host>:8002/sse
Authorization header: Bearer <your-api-key>
alwaysAllow: ["gcm_auth", "gcm_api", "gcm_discover"]

After writing the file, tell me what you changed and remind me to restart Bob.
```

Bob will read the file, add the entry, save it, and tell you to restart.

---

### B3. Option B — Configure Manually

Open the settings file (create it if it doesn't exist) and add the `gcm-mcp-server` block:

```json
{
  "mcpServers": {
    "gcm-mcp-server": {
      "type": "sse",
      "url": "http://<mcp-server-host>:8002/sse",
      "headers": {
        "Authorization": "Bearer <your-api-key>"
      },
      "alwaysAllow": ["gcm_auth", "gcm_api", "gcm_discover"]
    }
  }
}
```

> If the file already has other MCP servers, add the `gcm-mcp-server` block inside the existing `mcpServers` object — do **not** replace the whole file.

**Restart Bob** after saving.

---

### B4. Verify the Connection

After restarting Bob, ask:

> *"Use the GCM MCP server to list all available GCM services"*

Expected: Bob lists 11 services — usermanagement, tde, assetinventory, discovery, policy, policyrisk, audit, integration, notifications, clm, config.

## Part E: Key Rotation & Revocation (Admin)

### List all active keys

```bash
ssh root@<mcp-server-host>
curl -s http://localhost:8002/admin/keys
```

Output:

```json
[
  {"user": "alice@ibm.com", "key_prefix": "a3f8e9b1", "created": "2026-02-21T10:00:00Z"},
  {"user": "bob@ibm.com",   "key_prefix": "c5d2f7e3", "created": "2026-02-21T10:05:00Z"}
]
```

### Revoke a key immediately

```bash
curl -s -X DELETE http://localhost:8002/admin/keys/a3f8e9b1
```

The key is invalidated instantly — no container restart needed.

### Generate a replacement key

```bash
curl -s -X POST http://localhost:8002/admin/keys \
  -H "Content-Type: application/json" \
  -d '{"user": "alice@ibm.com"}'
```

Send the new key to the user and ask them to update their Bob config (replace the old key in the `Authorization` header).
